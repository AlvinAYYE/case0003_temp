## 1. 文件目的與讀者

本文將 RFP 與 LINE 討論內容整理成第一階段可開發、可驗收、可追蹤的軟體需求。文件主要讀者是具備前端與後端開發經驗，但尚未熟悉 LINE Official Account、LIFF、診所營運系統整合、GCP VM 維運與簡訊 OTP 串接的開發者。

本文刻意避免把未確認事項寫成既定事實。所有會影響範圍、估工、資料責任或維運責任的內容，若目前仍缺正式來源，會列在「待確認事項」。

## 2. 名稱與系統前提

RFP 與既有 API 文件中曾出現「悅容」與「悅康」兩種名稱。第一階段需求文件一律使用「第三方診所系統」作為暫稱，直到窗口確認正式產品名稱、廠商名稱、API base URL、API 文件版本與可用環境。

本系統暫稱為「診所 LINE 預約系統」。第一階段以 LINE Official Account 與 LIFF 手機版網頁作為主要入口，協助使用者完成會員綁定、本人或非本人預約、我的預約查詢、LINE 通知與行前提醒。

## 3. 第一階段目標

第一階段目標是建立一套可上線使用的最小完整預約流程：

1. 使用者從 LINE 官方帳號選單進入 LIFF 預約頁。
2. 系統透過 LINE Login / LIFF 取得並驗證 LINE 使用者身份。
3. 使用者完成會員綁定；若第三方診所系統查無客戶，系統可建立客檔。
4. 使用者可選擇本人或非本人預約。
5. 系統依分店、療程、醫師、排班、既有預約與療程規則計算可預約時段。
6. 使用者送出預約後，系統將正式預約寫入第三方診所系統。
7. 第三方診所系統回傳寫入成功時，本系統視為預約成功。
8. 系統發送 LINE 預約成功通知，並建立行前提醒排程。
9. 使用者可在「我的預約」查詢第三方診所系統名下的預約資料。
10. 診所人員可使用最小後台查詢會員、查看排班、設定療程規則與管理後台帳號。

## 4. 第一階段範圍

### 4.1 LINE 與入口設定

| 功能 | 需求 | 驗收方向 |
| --- | --- | --- |
| LINE Developers Provider | 建立或使用指定 Provider，完成基礎權限與安全設定 | 專案可管理 Messaging API、LINE Login、LIFF 與 Webhook |
| Messaging API Channel | 用於預約成功通知與行前提醒 | 可對測試 LINE user 發送訊息，並記錄成功或失敗 |
| LINE Login Channel | 用於取得可驗證的 LINE 使用者身份 | 後端可驗證登入結果，不信任前端直傳 UID |
| LIFF App | 提供手機版預約頁與我的預約頁 | 可從 LINE 內開啟，完成流程後可返回 LINE |
| Webhook | 接收加好友、封鎖等事件 | 事件可被接收、記錄，且重複事件不造成重複處理 |
| Rich Menu | 提供主要入口 | 至少包含加入/綁定會員、開始預約、我的預約 |

### 4.2 會員綁定與客檔建立

會員綁定用於建立 LINE user、手機號碼、本系統會員資料與第三方診所系統客戶資料之間的關聯。

第一階段流程：

1. 使用者從 LINE Rich Menu 點選加入/綁定會員。
2. LIFF 初始化並取得 LINE 登入狀態。
3. 後端驗證 LINE 身份並取得 LINE User ID。
4. 系統檢查本地是否已有啟用中的會員綁定。
5. 未綁定者填寫姓名、性別、生日、手機號碼。
6. 手機號碼格式需符合台灣手機格式：`09` 開頭，共 10 碼。
7. 使用者完成三竹簡訊 OTP 驗證。
8. 系統以手機號碼查詢第三方診所系統客戶資料。
9. 若找到既有客戶，建立本地會員綁定與第三方客戶 ID 對照。
10. 若找不到既有客戶，系統在第三方診所系統建立客檔，再建立本地綁定。

客檔建立失敗時，系統不得建立可用綁定；需向使用者顯示失敗狀態，並保存錯誤原因供後台或維運查詢。

### 4.3 三竹簡訊 OTP

第一階段簡訊僅用於手機驗證，不做大量行銷或其他簡訊通知。

| 項目 | 第一階段需求 |
| --- | --- |
| 供應商 | 三竹簡訊 |
| 發送用途 | 會員綁定手機驗證、非本人預約手機驗證 |
| API 形式 | 依 RFP 截圖，預期為 `SmsSend`，POST 至三竹網域下 `/b2c/mtk/SmSend` |
| 必要參數 | 帳號、密碼、收件手機、收件人名稱、簡訊內容、預約發送時間、有效期限等，以正式文件為準 |
| 編碼 | 需處理 URL Encode 與 `CharsetURL`；正式採 Big5 或 UTF-8 需待帳號文件確認 |

OTP 預設安全規則：

| 規則 | 預設值 |
| --- | --- |
| 重新發送冷卻 | 60 秒 |
| 驗證碼有效時間 | 5 分鐘 |
| 單一手機每日發送上限 | 5 次 |
| 單一 IP 每日發送上限 | 50 次 |
| 驗證錯誤鎖定 | 同一驗證流程錯誤 5 次後鎖定 |

系統需保存 OTP 發送紀錄、驗證結果、失敗原因、來源 IP、遮蔽後手機號碼與供應商回應摘要。不得在 log 中輸出完整驗證碼或完整手機號碼。

### 4.4 本人與非本人預約

預約流程分為本人預約與非本人預約。

本人預約：

1. 系統使用目前 LINE 帳號綁定的第三方客戶 ID。
2. 系統查詢該客戶的歷史預約、療程紀錄與目前預約。
3. 使用者選擇分店、療程、是否指定醫師與時段。
4. 系統寫入第三方診所系統預約。

非本人預約：

1. 使用者選擇非本人預約。
2. 使用者填寫被預約人姓名、生日與手機號碼。
3. 被預約人手機需完成 OTP 驗證。
4. 系統以手機查詢第三方診所系統客戶資料。
5. 若查無客戶，系統在第三方診所系統建立客檔。
6. 系統使用被預約人的第三方客戶 ID 建立預約。
7. 本系統需保存「由哪個 LINE 帳號代為建立」的操作紀錄，供後續客服與稽核查詢。

### 4.5 預約流程

核心預約流程：

1. 使用者點選開始預約。
2. 系統檢查 LINE 身份與會員綁定。
3. 使用者選擇本人或非本人預約。
4. 系統取得或建立預約對象的第三方客戶 ID。
5. 使用者選擇分店。
6. 系統讀取該分店可預約療程清單。
7. 系統依療程規則、客戶歷史療程與安全間隔過濾療程。
8. 使用者選擇療程。
9. 使用者選擇是否指定醫師；若指定，需選擇醫師。
10. 系統依療程耗時、醫師排班、既有預約與限制條件計算可用時段。
11. 使用者選擇時段。
12. 系統顯示預約摘要與術前提醒。
13. 使用者點選立刻預約。
14. 前端鎖定送出按鈕，後端以冪等機制避免重複建立預約。
15. 後端呼叫第三方診所系統建立正式預約。
16. 第三方診所系統回傳成功後，本系統建立預約快照與行前提醒排程。
17. 使用者看到預約成功畫面，並收到 LINE 預約成功通知。

第一階段預約成功定義：

- 第三方診所系統建立預約成功，即視為第一階段的預約成功。
- 若第三方診所系統回傳時段衝突、客戶資料錯誤、驗證失敗或其他錯誤，本系統需顯示失敗結果並保存錯誤紀錄。
- 本系統不得只因本地資料寫入成功就顯示預約成功。

### 4.6 可預約時段與療程規則

RFP 中的主要業務複雜度在於可預約時段不是單純列出空白時間，而是需要結合療程耗時、醫師排班、既有預約、客戶歷史與安全間隔進行計算。

運算輸入：

- 第三方客戶 ID。
- 分店 ID。
- 療程 ID。
- 療程耗時。
- 是否指定醫師。
- 指定醫師 ID。
- 查詢時間區間，預設以使用者目前瀏覽底點往後 7 天。
- 第三方診所系統中的醫師排班、休診、已預約資料。
- 本系統後台設定的療程先決條件與安全間隔。
- 客戶歷史預約與療程紀錄。

後台療程規則第一階段需支援：

| 規則 | 說明 |
| --- | --- |
| 療程耗時 | 以 15 分鐘為單位，例如 30、45、60 分鐘 |
| 先決療程 | 可設定某療程必須先完成其他療程 |
| 安全間隔 | 可設定距離前次療程需間隔的天數或分鐘數 |
| 術前提醒 | 可設定預約確認頁顯示的注意事項文字 |

系統應回傳可供前端顯示的不可預約原因，例如未滿足先決療程、安全間隔不足、無可用醫師、無連續空檔或第三方資料不足。

### 4.7 我的預約

第一階段需提供「我的預約」功能。

需求：

1. 使用者從 LINE Rich Menu 或 LIFF 內進入我的預約。
2. 系統驗證 LINE 身份並取得本地會員綁定。
3. 系統以綁定的第三方客戶 ID 查詢第三方診所系統名下的全預約。
4. 查詢結果需至少顯示分店、療程、醫師、日期時間與預約狀態。
5. 本頁只做單次頁面暫存；使用者重新進入頁面或手動刷新時，需重新向第三方診所系統查詢。

第一階段不提供取消與改期操作。若第三方診所系統回傳的預約資料包含已取消或已改期狀態，本系統可唯讀呈現，但不提供操作入口。

### 4.8 LINE 通知與行前提醒

預約成功通知：

- 觸發時機：第三方診所系統建立預約成功後。
- 發送對象：建立預約的 LINE 帳號。
- 內容至少包含分店、療程、醫師、日期時間、預約對象。
- 發送需非同步執行，避免 LINE API 失敗影響預約結果。

行前提醒：

- 由雲端排程或 VM 上的排程機制每日固定時間觸發。
- 查詢隔日預約，必要時向第三方診所系統校驗狀態。
- 每批次預設 50 筆，批次間隔 2 秒。
- LINE user 封鎖官方帳號或發送失敗時，需記錄失敗原因，不中斷整批任務。

### 4.9 最小後台

第一階段後台採最小可用範圍，不做完整 CRM、病歷、財務或庫存管理。

| 模組 | 第一階段需求 |
| --- | --- |
| 安全認證 | 管理員登入、登出、Session 逾時 |
| 權限角色 | Admin 可管理帳號與療程規則；Staff 可查會員與看排班 |
| 帳號管理 | Admin 可新增、停用、重設 Staff 密碼 |
| 會員查詢 | 依姓名、手機、LINE 綁定狀態搜尋會員 |
| 會員詳情 | 顯示本地會員資料、匯入資料摘要、第三方客戶 ID 與預約查詢入口 |
| 排班唯讀 | 選擇分店與醫師後，從第三方診所系統讀取排班並顯示 |
| 療程規則 | 設定耗時、先決療程、安全間隔與術前提醒 |
| 系統紀錄 | 查詢 OTP、LINE 通知、預約寫入失敗與第三方 API 錯誤摘要 |

## 5. 既有資料匯入

RFP 提到需處理約 5,000 筆既有客戶資料。第一階段採「本系統完整保存客戶基本資料、歷史預約與療程紀錄」的策略，但不納入病歷、財務、照片或其他高敏感資料，除非後續另行確認。

匯入範圍：

| 類型 | 第一階段保存內容 |
| --- | --- |
| 客戶基本資料 | 姓名、性別、生日、手機、第三方客戶 ID、來源資料識別 |
| 聯絡資訊 | 以預約與綁定必要欄位為限 |
| 歷史預約 | 分店、療程、醫師、時間、狀態、第三方預約 ID |
| 歷史療程 | 療程名稱或代碼、執行時間、必要狀態欄位 |
| 匯入稽核 | 來源檔名、匯入批次、成功/失敗筆數、錯誤原因 |

匯入驗收方向：

1. 可讀取指定來源檔並產生欄位對應報告。
2. 可偵測手機格式錯誤、重複客戶、缺漏必要欄位。
3. 可產生匯入前檢查報告，不直接覆蓋既有資料。
4. 匯入後可依手機或姓名查到客戶資料。
5. 匯入後療程規則可使用歷史療程紀錄判斷先決條件與安全間隔。

## 6. 外部整合介面

### 6.1 第三方診所系統 API

第一階段需要的能力：

| 類型 | 需求 |
| --- | --- |
| 授權 | 取得與刷新 API token，管理憑證與錯誤 |
| 客戶 | 依手機查詢客戶、建立客戶、讀取客戶基本資料 |
| 分店與醫師 | 讀取分店、醫師、科別或相關主檔 |
| 療程 | 讀取分店可預約療程或服務項目 |
| 排班 | 讀取指定時間區間內醫師排班、休診與可用資訊 |
| 預約 | 建立預約、查詢預約列表、查詢單筆預約 |
| 事件 | 若第三方系統支援 webhook 或 event history，需評估是否用於補償與同步 |

第一階段不得讓第三方 API schema 直接滲透到前端或核心 domain。後端需透過 adapter 將第三方資料轉為本系統 DTO 或 domain object。

### 6.2 LINE API

第一階段使用：

- LINE Login / LIFF：驗證使用者身份。
- Messaging API：發送預約成功與行前提醒。
- Webhook：接收加好友、封鎖等事件。
- Rich Menu：提供主要 LIFF 入口。

### 6.3 三竹簡訊

第一階段使用三竹簡訊發送 OTP。帳號、密碼、正式網域、測試方式、費用儲值與企業申請資料需由專案窗口確認。

## 7. 核心資料責任

| 資料 | 本系統是否保存 | 說明 |
| --- | --- | --- |
| LINE User ID | 是 | 會員綁定與通知必要資料 |
| 會員綁定 | 是 | 保存 LINE user、手機、本地會員、第三方客戶 ID 對照 |
| 手機號碼 | 是 | OTP、綁定、查詢與客服需要；log 需遮蔽 |
| OTP 紀錄 | 是 | 防刷、稽核與錯誤排查 |
| 匯入客戶基本資料 | 是 | 依第一階段匯入範圍保存 |
| 匯入歷史預約與療程 | 是 | 支援療程規則、查詢與客服 |
| 第三方正式預約 | 第三方為主，本系統保存快照 | 預約成功後保存必要快照與第三方預約 ID |
| 我的預約查詢結果 | 不長期保存 | 單次頁面暫存；重新進入或刷新就重查 |
| 療程規則 | 是 | 本系統後台設定與時段計算需要 |
| LINE 通知紀錄 | 是 | 發送狀態、失敗原因與重試 |
| 行前提醒排程 | 是 | 排程掃描、發送與補償需要 |
| 管理員帳號 | 是 | 後台登入與權限管理 |
| 病歷、財務、照片 | 否 | 不列入第一階段，除非後續另行確認 |

## 8. 部署與環境

第一階段採 GCP VM + Docker 部署應用程式容器，資料庫使用 Cloud SQL。

部署需求：

| 項目 | 第一階段需求 |
| --- | --- |
| 運算主機 | GCP VM |
| 容器化 | 使用 Docker 部署前端、後端與必要 worker |
| 資料庫 | Cloud SQL |
| 排程 | 使用 VM Cron、Cloud Scheduler 或等效方式觸發行前提醒 |
| 網路 | DNS、Static IP、HTTPS / SSL |
| 憑證 | LINE、第三方診所系統、三竹簡訊憑證以環境變數或 Secret 管理 |
| 備份 | Cloud SQL 需設定備份策略 |
| Log | 至少保留後端錯誤、第三方 API 錯誤、簡訊與 LINE 發送紀錄 |

若第三方診所系統要求固定出口 IP 或 IP 白名單，需將 GCP VM 的固定外部 IP 或 NAT 設定納入正式部署驗收。

## 9. 非目標

第一階段不包含：

- 原生 iOS / Android App。
- 取消預約操作。
- 改期操作。
- 後台直接編輯第三方診所系統排班。
- 完整 CRM、病歷、財務、庫存系統。
- 病歷、財務、照片資料匯入。
- 多診所集團層級權限。
- 多語系。
- 大量行銷訊息或促銷推播。
- LINE 以外的登入方式。
- 除 OTP 外的大量簡訊通知。

## 10. 風險與處理方向

| 風險 | 影響 | 處理方向 |
| --- | --- | --- |
| 第三方 API 名稱與版本未確認 | 文件、程式與溝通可能混亂 | 正式開發前確認名稱、base URL、版本與 sandbox |
| 第三方 API 寫入失敗或回應不穩 | 直接影響預約成功率 | 優先 PoC 授權、建客檔、建預約、查預約 |
| 預約並發搶位 | 同一時段可能被多人同時送出 | 前端鎖按鈕、後端冪等、依第三方錯誤處理衝突 |
| 療程規則口徑不清 | 可能算出診所不接受的時段 | 請診所確認耗時、先決條件、安全間隔與範例 |
| 既有資料品質不佳 | 匯入錯誤會影響綁定與療程判斷 | 匯入前產生檢查報告與錯誤清單 |
| 個資保存範圍變大 | 資安與維運責任增加 | 權限控管、遮蔽、備份、最小 log、匯入稽核 |
| LINE 或簡訊發送失敗 | 使用者收不到通知或 OTP | 保存供應商回應、錯誤碼與重試策略 |
| VM 維運責任 | 需要處理更新、SSL、監控與備份 | 明確分配維運責任並建立基本操作文件 |
