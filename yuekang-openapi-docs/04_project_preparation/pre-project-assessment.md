# API 對接前期評估報告

## 1. 結論先行

這套 API 可判定為中大型業務整合型 SaaS 平台 API。它不只是 CRUD，而是覆蓋院所營運、客戶、產品、診療、銷售財務、庫存、SCRM、電子健康檔案與事件訂閱的開放平台。

在需求尚未落地前，不建議直接開正式功能。先做 PoC 與需求收斂，確認資料邊界、主系統、同步頻率與失敗補償。

## 3. 可能落地場景

| 場景 | 可行性 | 需要驗證 |
| --- | --- | --- |
| 客戶資料同步到 CRM/CDP | 高 | 唯一鍵、欄位、更新與刪除語意 |
| 訂單/收款/退款同步到財務系統 | 高，但風險較高 | 對帳口徑、時間口徑、權益消耗 |
| 預約、到診、診療狀態同步 | 中高 | 狀態流轉與外部是否可寫入 |
| 庫存同步到 ERP/WMS | 中高 | 入出庫作廢、調撥與庫存一致性 |
| 事件訂閱近即時同步 | 高，但必須 PoC | callback、重試、簽名、冪等、保序與補償 |

## 4. 最大不確定點

- 尚不知道實際要落地的業務目標。
- 文件未說明 webhook callback 的簽名、重試、超時與保序策略。
- 不確定正式環境 rate limit、IP 白名單與 token 有效期。
- 歷史事件僅支援近 3 個月，首次全量同步需另定方案。

## 5. 建議下一步

1. 選 2-3 個最可能落地的場景。
2. 做 3-5 天 PoC，驗證 token、查詢、分頁、topic、訂閱與歷史補償。
3. 將對接問題清單交給 API 提供方確認。
4. 依 PoC 結果決定正式開發範圍與架構。
